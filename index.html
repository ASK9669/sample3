Introduction to Data Modeling

Data modeling is the process of creating an abstract representation of the data stored in a database. Before constructing any database, designers must first conceptualize the information structure using a high-level model that all stakeholders — including non-technical users — can understand.

The Entity-Relationship (ER) Model, introduced by Peter Chen in 1976, is the most widely used conceptual data model. It provides a graphical notation (ER Diagrams) to describe the structure of a database in terms of entities, attributes, and relationships.

🎯 Core Purpose The ER model acts as a communication tool between database designers and end users. It captures real-world entities and the associations between them before any physical implementation begins.

Database Design Levels

🗺️

Conceptual Level

High-level description of data using ER diagrams. Technology-independent view of the data.

🏗️

Logical Level

ER model is mapped to a relational schema (tables, columns, keys). Still independent of DBMS.

💾

Physical Level

Actual storage structures, indexes, and file organization. DBMS-specific implementation.

📋 Why ER Modeling?
  • ✓ Provides clear documentation of database structure
  • ✓ Facilitates communication with end users
  • ✓ Helps identify data requirements early
  • ✓ Independent of any specific DBMS
  • ✓ Reduces implementation errors
  • Data modeling creates an abstract representation of a real-world domain's data.
  • The ER Model (Chen, 1976) is the most popular conceptual model for database design.
  • Design moves through three levels: Conceptual → Logical → Physical.
  • ER diagrams serve as a universal communication tool between designers and users.
  • Conceptual design is technology-independent; physical design is DBMS-specific.

Entities & Entity Types

An entity is any object or concept in the real world that exists independently and has data worth storing. Examples include a person, a car, a company, or a bank account.

An entity type defines a collection (set) of entities that share the same attributes. It acts as a blueprint. An entity instance is a specific occurrence — e.g., "Employee John Smith" is an instance of the EMPLOYEE entity type.

ER Notation: Entity Symbols

Strong Entity

  • Exists independently
  • Has its own primary key
  • Drawn as a single rectangle
  • e.g., EMPLOYEE, STUDENT, PRODUCT

Weak Entity

  • Cannot exist without an owner entity
  • No primary key of its own
  • Drawn as a double rectangle
  • e.g., DEPENDENT (of EMPLOYEE)
📝 Key Term: Partial Key A weak entity has a partial key (underlined with dashed line in ER diagrams) — an attribute that uniquely identifies weak entity instances relative to their owner. For example, a DEPENDENT is identified by (employee_id + dependent_name).
  • An entity is a real-world object or concept with data to store.
  • An entity type = blueprint (class); entity instance = specific occurrence.
  • Strong entities exist independently and have their own primary key.
  • Weak entities depend on an owner entity and use a partial key for identification.
  • Strong entity → single rectangle; Weak entity → double rectangle.

Attributes & Their Types

An attribute is a property or characteristic of an entity type or relationship type. Each attribute has a domain — the set of allowed values. For example, the EMPLOYEE entity might have attributes: Name, Age, Salary, Address.

Cannot be subdivided into smaller meaningful parts.

Example: Age, Salary, Employee_ID

ER Symbol: Plain oval (ellipse)

Can be divided into smaller subparts, each with independent meaning.

Example: Name → {First_Name, Middle_Initial, Last_Name} | Address → {Street, City, State, Zip}

ER Symbol: Oval connected to sub-ovals

Can hold multiple values for a single entity instance.

Example: Phone_Numbers (a person may have home, mobile, work) | College_Degrees

ER Symbol: Double oval (ellipse with double border)

Its value can be computed from another attribute — not stored directly.

Example: Age derived from Birth_Date | Years_Employed derived from Hire_Date

ER Symbol: Dashed oval

Uniquely identifies each entity instance in an entity type. No two instances can share the same key value.

Example: SSN for EMPLOYEE | VIN for CAR | Student_ID for STUDENT

ER Symbol: Oval with underlined text

An attribute may have a NULL value when it is inapplicable or unknown for a particular entity. NULL is not zero or blank — it means "no value."

Example: Apartment_Number is null if the employee lives in a house.

ER Diagram: Attribute Notation

Attribute Type Description ER Symbol Example
Simple Indivisible, single-valued Plain oval Salary, Age
Composite Divisible into sub-parts Oval with sub-ovals Name, Address
Multi-valued Multiple values per entity Double oval Phone_Numbers
Derived Computed from another attribute Dashed oval Age (from DOB)
Key Uniquely identifies instances Underlined oval SSN, Student_ID
  • Attributes describe properties of entity types and relationship types.
  • Simple attributes are atomic; composite attributes have meaningful sub-parts.
  • Multi-valued attributes store multiple values (e.g., phone numbers) — shown as double oval.
  • Derived attributes are calculated, not stored — shown as dashed oval.
  • Key attributes uniquely identify each entity instance — underlined in ER diagrams.
  • NULL means a value is missing or inapplicable, not zero or blank.

Relationships & Relationship Types

A relationship is an association between two or more entities. A relationship type defines the set of associations among entity types. In ER diagrams, relationships are depicted as diamond shapes connected to the participating entity types.

The degree of a relationship is the number of entity types participating in it.

⬡

Unary (Degree 1)

A relationship between instances of the same entity type. Also called a recursive relationship. e.g., EMPLOYEE supervises EMPLOYEE.

⬡

Binary (Degree 2)

Between two different entity types. Most common. e.g., EMPLOYEE works_on PROJECT.

⬡

Ternary (Degree 3)

Involves three entity types simultaneously. e.g., SUPPLIER supplies PART to PROJECT.

ER Diagram: Binary Relationship Example

🔗 Relationship Attributes Relationships can have their own attributes. For example, the WORKS_ON relationship between EMPLOYEE and PROJECT can have an attribute Hours_Per_Week — how many hours an employee works on a specific project.

Role Names

In recursive (unary) relationships, role names clarify how each entity instance participates. For example, in the EMPLOYEE supervises EMPLOYEE relationship, one role is supervisor and the other is subordinate.

  • Relationships represent associations between entities — drawn as diamonds in ER diagrams.
  • Degree of a relationship = number of participating entity types (unary, binary, ternary).
  • Binary relationships (degree 2) are the most common in practice.
  • Relationships can have their own attributes (e.g., Hours_Per_Week on WORKS_ON).
  • Unary/recursive relationships use role names to distinguish each entity's role.

Relationship Constraints

Relationship constraints specify rules about how entities can participate in relationships. There are two main types: cardinality ratios and participation constraints.

Cardinality Ratios

A cardinality ratio specifies the maximum number of relationship instances an entity can participate in. The three basic types for binary relationships are:

Participation Constraints

Participation constraints specify the minimum number of relationship instances an entity must participate in — in other words, whether participation is mandatory or optional.

Total Participation (Mandatory)

  • Every entity must participate in at least one relationship instance
  • Minimum cardinality = 1
  • Drawn as a double line to the relationship diamond
  • e.g., Every EMPLOYEE must work in a DEPARTMENT

Partial Participation (Optional)

  • Some entities may NOT participate in any relationship instance
  • Minimum cardinality = 0
  • Drawn as a single line to the relationship diamond
  • e.g., Not every EMPLOYEE manages a DEPARTMENT
📐 Min-Max Notation The (min, max) notation is more expressive than simple 1:N ratios. It specifies both minimum and maximum participations.

Example: EMPLOYEE (1,1) ─── works_in ─── (1,N) DEPARTMENT
Means: Each employee works in exactly 1 department; each department has 1 or more employees.
  • Cardinality ratio = maximum number of relationships an entity can participate in (1:1, 1:N, M:N).
  • Participation constraint = minimum participation: total (mandatory, double line) vs partial (optional, single line).
  • 1:1 example: EMPLOYEE manages DEPARTMENT (one manager per dept).
  • 1:N example: DEPARTMENT has EMPLOYEES (many employees per dept).
  • M:N example: EMPLOYEE works on PROJECTS (many-to-many).
  • Min-Max notation (min,max) gives precise structural constraints.

Weak Entity Types & Identifying Relationships

A weak entity type does not have a key attribute of its own. It is identified in the context of its owner entity type through an identifying relationship.

Weak entities always have total participation in their identifying relationship — they cannot exist without their owner.

ER Diagram: Weak Entity — DEPENDENT of EMPLOYEE

📌 Key Rules for Weak Entities
  • ✓ Drawn as a double rectangle
  • ✓ Identified by its owner's key + its own partial key
  • ✓ The identifying relationship is drawn as a double diamond
  • ✓ Always has total participation in the identifying relationship
  • ✓ Partial key is underlined with a dashed line
  • Weak entities have no primary key of their own.
  • They are identified by combining the owner entity's key + the weak entity's partial key.
  • The relationship connecting a weak entity to its owner is called an "identifying relationship" — drawn as a double diamond.
  • Weak entities always have total (mandatory) participation in the identifying relationship.
  • Example: DEPENDENT (partial key = name) identified by EMPLOYEE (SSN) + DEPENDENT (name).

Enhanced ER (EER) Model

The Enhanced ER (EER) Model extends the basic ER model with additional semantic concepts borrowed from object-oriented design. These include generalization, specialization, inheritance, categories (union types), and aggregation.

⬆️

Generalization

Bottom-up process: Multiple entity types are combined into a higher-level (superclass) entity. e.g., CAR and TRUCK → VEHICLE.

⬇️

Specialization

Top-down process: A superclass is refined into more specific subclasses based on distinguishing characteristics. e.g., EMPLOYEE → SECRETARY, ENGINEER, MANAGER.

🧬

Inheritance

Subclasses inherit all attributes and relationships of their superclass, plus add their own specific attributes/relationships.

🔀

Category (Union Type)

A subclass of the union of two or more superclasses. e.g., OWNER can be a PERSON or a BANK or a COMPANY.

Specialization Constraints

Constraint Options Meaning
Disjointness Disjoint (d) An entity can belong to at most ONE subclass. e.g., A vehicle is either a Car OR a Truck, not both.
Disjointness Overlapping (o) An entity can belong to MULTIPLE subclasses simultaneously. e.g., A person can be both a Student and an Employee.
Completeness Total Every superclass entity MUST belong to at least one subclass. e.g., Every SHAPE must be a circle, rectangle, or triangle.
Completeness Partial Superclass entities may NOT belong to any subclass. e.g., An EMPLOYEE may not be classified as SECRETARY or ENGINEER.

Aggregation

📦 Aggregation Aggregation treats a relationship and its participating entities as a higher-level entity. Used when a relationship itself needs to participate in another relationship. Example: The WORKS_ON relationship between EMPLOYEE and PROJECT can be aggregated into a whole entity that is then related to EQUIPMENT.
  • EER extends ER with OOP-like concepts: specialization, generalization, inheritance, categories, aggregation.
  • Specialization = top-down refinement (superclass → subclasses).
  • Generalization = bottom-up abstraction (subclasses → superclass).
  • Subclasses inherit all attributes and relationships of their superclass.
  • Disjoint (d): one subclass only. Overlapping (o): can belong to multiple subclasses.
  • Total completeness: every superclass entity must be in a subclass. Partial: optional.
  • Aggregation wraps a relationship as an entity to allow it to participate in other relationships.

ER Design Process & Guidelines

Creating a good ER schema requires systematic analysis of requirements and careful design decisions. The typical process follows these phases:

1. Requirement Collection & Analysis

Interview users, study existing forms/reports, identify data requirements. Produce a data requirements document describing all entities, relationships, and constraints.

2. Conceptual Database Design

Construct a high-level ER diagram using the gathered requirements. Identify entity types, relationships, attributes, keys, and constraints.

3. Logical Database Design (Data Model Mapping)

Convert the ER schema to a relational schema (tables). Map entities → tables, attributes → columns, keys → primary/foreign keys, M:N relationships → junction tables.

4. Physical Database Design

Specify storage structures, indexes, access paths, and file organizations. This is DBMS-specific and aims for performance optimization.

ER Design Guidelines: Common Pitfalls

Occurs when a model represents relationships between entity types that are incorrectly mapped, causing ambiguity in traversal paths. Happens when two 1:M relationships radiate from the same entity but their combination is ambiguous.

Solution: Re-analyze the real-world relationship and restructure the ER diagram.

Occurs when a path exists between entity types but for some entities no path exists due to partial participation. There is a "chasm" or gap in the data.

Solution: Review participation constraints and ensure that the correct entities are connected.

A common design question: should something be an entity or an attribute? If it has multiple attributes of its own, or participates in relationships independently, it should be an entity. Otherwise, an attribute may suffice.

Example: "Department" with its own name, location, and budget → entity type; "Salary" as a single value → attribute of EMPLOYEE.

If an association has its own attributes or participates in other relationships, consider making it an associative (weak) entity instead of a relationship.

Example: "ENROLLMENT" between STUDENT and COURSE has grade and date attributes → better as an associative entity.

ER → Relational Mapping Summary

ER Concept Maps To (Relational)
Strong Entity Type Table (relation) with a primary key
Simple Attribute Column in the entity's table
Composite Attribute Multiple columns (one per component)
Multi-valued Attribute Separate table + foreign key to owner
Derived Attribute Usually NOT stored; computed in queries
Weak Entity Type Table with composite PK (owner PK + partial key)
1:1 Relationship FK in one table (or merge tables)
1:N Relationship FK on the N-side entity's table
M:N Relationship Separate junction/association table
Specialization/Generalization Multiple approaches: single table, separate tables, or hybrid
  • Database design: Requirements → Conceptual (ER) → Logical (Relational) → Physical (DBMS-specific).
  • Fan trap & chasm trap are common ER design mistakes from incorrect relationship modeling.
  • Decide entity vs attribute: if it has its own attributes or relationships, make it an entity.
  • Strong entities → tables; simple attributes → columns; multi-valued attributes → separate tables.
  • 1:N relationships → FK on the many-side; M:N → junction table; weak entities → composite PK.
